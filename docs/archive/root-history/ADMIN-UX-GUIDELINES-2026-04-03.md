# Admin UX Guidelines

## Obiettivo

Riportare `account` e `amministrazione` su una UX piu' chiara, moderna e coerente con una piattaforma di spedizioni che fa da intermediario.

## Principi

1. `Admin` e `Account` non sono la stessa superficie.
2. `Partner Pro` e' un'area funzione, non uno switch di ruolo.
3. La root admin deve servire l'operativita', non il rumore.
4. Pochi blocchi forti battono molte card equivalenti.
5. Colore semantico e sobrio, non decorativo.
6. Testo minimo, ma non testo zero.
7. Un punto focale chiaro per schermata.

## Root Admin

La root admin deve mostrare soprattutto:

- KPI essenziali
- code urgenti
- un focus visivo forte
- accessi rapidi ai moduli core

I moduli core sono:

- Ordini
- Tracking
- Prelievi
- Messaggi
- Utenti

La root admin non deve diventare:

- sitemap del backoffice
- collage di widget
- pagina piena di testo introduttivo

## Root Account

La root account deve mostrare:

- aree personali vere
- un ingresso chiaro a `Partner Pro`
- un solo ingresso alla `Console admin` per gli admin

Non deve mostrare:

- tab che sembrano cambiare identita'
- copie integrali del backoffice dentro l'area personale
- troppi modi diversi per arrivare allo stesso posto

## Copy

- titolo breve
- label chiara
- micro-descrizione solo se orienta davvero
- niente testo lungo per spiegare una card

## Colore

- teal / brand per azione primaria e flusso normale
- amber per attenzione
- red per criticita'
- green per positivo / saldo / esito buono
- neutri per contenuto standard

## Riferimenti Esterni Filtrati

- Sendcloud: ordini e processo operativo come centro del workspace
- Easyship: flusso spedizioni e azioni chiare prima della decorazione
- Shippo: ordini trattati come record di lavoro
- Figma IA: gerarchia e wayfinding prima della grafica
- Baymard: densita' controllata, poco rumore, stati leggibili

## Moduli Prioritari

1. `/account/amministrazione/ordini`
2. `/account/amministrazione/prelievi`
3. `/account/amministrazione/portafogli`
4. funnel step `2/3`
