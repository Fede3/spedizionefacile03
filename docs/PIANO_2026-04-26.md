## 1. Filosofia: no rompere nulla

Tre regole operative non negoziabili:

1. **Test verde a ogni commit**. Prima di iniziare la fase 0 si tagga
   `pre-refactor-2026-04-26` come baseline. Ogni step successivo deve passare
   `npm run build` + `php artisan test` + `npx playwright test`. Se rompe
   qualcosa, si revert immediato e si splitta lo step.
2. **Una fase per volta**. Non incrociare interventi (es. non spostare CSS
   mentre si splitta un composable). Ogni fase dura 3-7 giorni e ha un PR
   chiuso prima della successiva.
3. **Strangler pattern**, non rewrite. Quando si splitta un god composable in
   store Pinia atomici, lo si fa **affianco al vecchio**, si testa, si redirige
   il consumer al nuovo, poi si elimina il vecchio. Mai sostituire in place.

Conseguenza: il prodotto resta sempre funzionante. In ogni momento si può
mettere il refactor in pausa per priorità di business.

### Fase 2 — CSS razionalizzato (2-3 settimane) ★ priorità più alta dopo cleanup

**Obiettivo**: 66 file CSS → 10-12 file. ~24.700 LOC CSS → ~6.000 LOC.
Adottare Tailwind 4 utility-first come pattern dominante.

**Perché è il refactor con il payoff più alto**:
1. Riduce LOC del **17% globale** (-18.000 LOC).
2. Elimina 50+ duplicazioni layout/colore/typography.
3. Rende ogni componente **leggibile** (lo stile è inline alle classi, non
   sparso in 4 file CSS).
4. Junior trova lo stile di un bottone in 5 secondi (`grep "rounded-xl"` nel
   componente), non in 1 ora.

**Strategia incrementale per non rompere nulla**:

#### 2.1 Setup nuova CSS architecture (1 giorno)

Creare nuova struttura **affianco** alla vecchia:
```
assets/css/
  v2/
    tokens.css       # CSS variables (colori, spacing, font)
    base.css         # reset + typography globale
    components.css   # utility custom (.sf-button, .sf-card, .sf-segment)
  [vecchi 66 file restano, finché migrazione non finita]
```

**`tokens.css`** centralizza TUTTI i colori e spacing. Esempio:
```css
:root {
  --color-brand-teal: #095866;
  --color-brand-orange: #E44203;
  --color-brand-teal-dark: #064551;
  /* etc. */
}
```

**`components.css`** contiene SOLO le ~10 utility custom davvero riusabili:
```css
.sf-button {
  @apply inline-flex items-center gap-2 px-6 py-3 rounded-xl font-semibold
         transition-colors duration-200 focus-visible:ring-2;
}
.sf-card { @apply rounded-2xl border bg-white shadow-sm p-6; }
.sf-segment { ... }
.sf-form-field { ... }
```

#### 2.2 Migrazione pagina-per-pagina (gran parte del tempo)

**Ordine consigliato** (dal più semplice al più complesso):
1. `pages/contatti.vue`, `faq.vue`, `chi-siamo.vue` (pagine statiche).
2. Componenti footer/header.
3. Pagine auth (`login`, `registrazione`, `recupera-password`).
4. Account dashboard.
5. Admin (admin-prezzi, admin-ordini, admin-utenti).
6. Checkout/funnel (per ultimo, è il più rischioso).

Per ogni pagina:
1. **Identificare** classi CSS custom usate (grep nei file `.vue` della cartella).
2. **Mappare** ognuna a equivalente Tailwind utility o utility custom in
   `components.css`.
3. **Riscrivere** il template con classi Tailwind inline.
4. **Eliminare** il blocco `<style>` del componente o il file CSS dedicato.
5. **Verificare visivamente** la pagina (preview MCP, screenshot before/after).
6. **Commit atomico** per pagina: `refactor(css): migrate contatti to Tailwind`.

#### 2.3 Eliminazione vecchi CSS (gradualmente)

Quando una sezione è migrata, rimuovere il CSS dedicato e l'import in `main.css`.
Esempio: dopo migrato pagine auth, rimuovere `autenticazione.css` + import
relativo.

#### 2.4 Manutenzione `main.css`

Da 13 `@import` a cascata → 3:
```css
@import "v2/tokens.css";
@import "v2/base.css";
@import "v2/components.css";
```

**Risultato atteso**: 66 file CSS → 10-12. ~24.700 LOC → ~6.000.

**Rischi**:
- Visual regression. Mitigazione: screenshot before/after per pagina, test
  Playwright visual.
- Tempo. È il refactor più lungo. Si può **interrompere a metà** senza danno
  (la nuova struttura coesiste con la vecchia).

**Done quando**:
- < 15 file CSS in `assets/css/`.
- LOC CSS < 8.000.
- 0 `@import` in cascata in `main.css` (max 3 import diretti).
- Nessuna pagina ha visual regression vs baseline (screenshot diff verde).

### Fase 3 — Split god composables in Pinia store atomici (2 settimane)

**Obiettivo**: i 7 composables > 700 LOC diventano store Pinia atomici (200-400
LOC ognuno) o componenti figli.

**Mapping refactor**:

| Composable attuale | LOC | Diventa |
|---|---|---|
| `useAdminPrezzi.js` | 1.426 | `stores/adminPricingStore.js` (250) + `stores/adminSupplementsStore.js` (150) + componenti form/list/import inline pagina |
| `usePreventivo.js` | 1.326 | `stores/preventivoStore.js` (350) + 3 componenti `<PreventivoForm>`, `<PreventivoSummary>`, `<PreventivoActions>` |
| `usePudo.js` | 912 | `stores/pudoStore.js` (300) + componente `<PudoMap>`, `<PudoList>` |
| `useCart.js` | 820 | `stores/cartStore.js` (300) + componente `<CartItems>`, `<CartTotals>` |
| `useShipmentStepServices.js` | 754 | merge in `stores/shipmentFlowStore.js` (estensione) |
| `usePayment.js` | 745 | `stores/paymentStore.js` (250) + componente `<PaymentMethodSelector>` |
| `useFunnel.js` | 744 | merge in `stores/shipmentFlowStore.js` (estensione) |

#### 3.1 Strangler pattern per ogni composable (esempio `useCart`)

**Step 1 — Creare il nuovo store** `stores/cartStore.js`:
```js
import { defineStore } from 'pinia'

export const useCartStore = defineStore('cart', () => {
  const items = ref([])
  const total = computed(() => items.value.reduce((s, i) => s + i.price, 0))

  async function fetchItems() { /* logica da useCart */ }
  async function addItem(item) { /* logica da useCart */ }
  // ... 5-7 azioni totali

  return { items, total, fetchItems, addItem, ... }
})
```

**Step 2 — Lasciare `useCart.js` come deprecation shim** (zero-rischio durante
migrazione):
```js
// composables/useCart.js
import { useCartStore } from '~/stores/cartStore'
export function useCart() {
  console.warn('[deprecated] useCart() — use useCartStore() instead')
  return useCartStore()
}
```

**Step 3 — Migrare i consumer** uno per uno:
```diff
- const { items, addItem } = useCart()
+ const cart = useCartStore()
+ const { items } = storeToRefs(cart)
+ const { addItem } = cart
```

**Step 4 — Test pagina dopo ogni migrazione**.

**Step 5 — Eliminare `useCart.js`** quando 0 import.

#### 3.2 Ordine consigliato

Dal più isolato al più trasversale (per ridurre rischio):
1. `useAdminPrezzi` (usato solo in `admin/prezzi.vue`).
2. `usePudo` (usato solo in `pudo.vue`).
3. `useCart` (usato in `carrello.vue` + checkout step).
4. `usePreventivo` (usato in `Preventivo.vue` + step 1).
5. `usePayment` + `useShipmentStepServices` + `useFunnel` (estensioni
   `shipmentFlowStore`).

#### 3.3 Risultato atteso

- 40 composables → ~20.
- 7 god > 700 LOC → 0 god > 500 LOC.
- 3 stores Pinia → 7-8 stores atomici.
- ~7.500 LOC di composables → ~3.500 in stores + ~1.500 nei componenti =
  netto -2.500 LOC (eliminando duplicazioni emerse).

**Done quando**:
- 0 composables > 500 LOC.
- Nuovi store atomici documentati in `docs/STATE_MANAGEMENT.md` (aggiornare).
- Test E2E + manuale verde su tutte le pagine toccate.

## 6. Anti-pattern da evitare

Lista esplicita di cose che **possono sembrare buone idee** ma rovinerebbero
il refactor. Tenere a mente costantemente.

### ❌ Big bang rewrite

Riscrivere il frontend da zero "perché tanto è troppo complesso". **No.** Il
dominio è complesso, le integrazioni sono delicate, c'è già copertura test.
Ogni rewrite reintroduce bug già risolti.

### ❌ Mix di interventi nello stesso PR

Splittare un god composable mentre si migrano CSS. **No.** Un PR = una fase.
Se si rompe qualcosa non si capisce cosa.

### ❌ Refactor "estetico" senza test

"Cambio nome a 5 funzioni perché mi sembrano brutti". **No.** Solo refactor
strutturale con metrica obiettiva (LOC, complessità ciclomatica, file count).

### ❌ Add new feature mid-refactor

"Già che ci sono aggiungo questo nuovo metodo di pagamento". **No.** Il refactor
è zero-feature. Una feature in mezzo rischia di mascherare regressioni.

### ❌ Skip linting "perché tanto so cosa faccio"

I commit `--no-verify` durante il refactor sono vietati. Se pre-commit blocca,
si fixa il codice, non si bypassa.

### ❌ Refactor solo backend, lasciare CSS

Tentazione: il CSS è il refactor più lungo, "lo facciamo dopo". **No.** Il CSS
è il refactor con il **payoff più alto** sull'accessibilità junior. Va prima.

### ❌ Sed mass per defensive optional chaining

I commit recenti `fix(ssr): correggi assignment patterns rotti dal sed mass ?.`
sono il sintomo. Mai più. Se serve refactor sistematico si usa AST tool
(jscodeshift, ts-morph, php-parser) o si fa file-per-file con review.

### ❌ Pinia store per ogni cosa

Antitesi del "mono-store" è "150 store mono-uso". Il target è **6-8 store
atomici**, non 30. Composables esistono ancora per logica trasversale riusabile.

### ❌ CSS scoped sparso ovunque

In Vue/Nuxt `<style scoped>` è ok per stili davvero locali, ma se si trova lo
stesso `.button` in 10 componenti `<style scoped>` è duplicazione. Tornano
in `components.css` come utility custom.

## 7. Manutenzione continua dopo il refactor

Il refactor **non è una tantum**. Per non tornare a 110K LOC:

### 7.1 Regole architetturali da committare

Aggiungere a `CLAUDE.md` (root) e `docs/CONTRIBUTING.md`:

> **Limiti di dimensione file**:
> - File `.vue`: max **400 LOC**. Sopra → splittare in sub-component.
> - Composables `.js/.ts`: max **300 LOC**. Sopra → diventa Pinia store atomico.
> - Pinia store: max **400 LOC**. Sopra → splittare per dominio.
> - Controller PHP: max **200 LOC**. Sopra → estrarre Action class.
> - Service PHP: max **400 LOC**. Sopra → estrarre Value Object o splittare.
> - File CSS: max **400 LOC**. Sopra → splittare per concern.

### 7.2 Lint check su file size in CI

Aggiungere job CI `check-file-size`:
```yaml
- name: Check god files
  run: |
    GOD_FILES=$(find . -name '*.{vue,js,ts,php}' \
      -not -path './node_modules/*' \
      -not -path './vendor/*' \
      -exec wc -l {} + | awk '$1 > 500 && $2 != "total"')
    if [ -n "$GOD_FILES" ]; then
      echo "❌ God files detected (>500 LOC):"
      echo "$GOD_FILES"
      exit 1
    fi
```

### 7.3 Code review checklist

Aggiungere `.github/PULL_REQUEST_TEMPLATE.md`:
```markdown
## Checklist
- [ ] Nessun nuovo file > 400 LOC
- [ ] Nessun nuovo composable usato in una sola pagina (deve essere Pinia store o sub-component)
- [ ] Stili usano Tailwind utility o sf-* classes esistenti (no nuovo CSS custom senza ragione)
- [ ] Test E2E coprono la modifica
- [ ] `pint` + `eslint` + `prettier` verdi
```

### 7.4 Refactor budget

Ogni sprint riserva **1 giorno per refactor preventivo**. Se un file si avvicina
al limite (350 LOC), si splitta proattivamente prima che diventi god.

### 7.5 Onboarding junior come stress test

Quando si assume un junior, misurare tempo reale a essere produttivo. Se è
> 2 settimane, c'è degrado. Tornare al log fasi e capire dove si è regrediti.
